ReactJS Sans Tooling

The single biggest roadblock in the ReactJS ecosystem is the ovewhelming amount of tooling, additional libraries, & architechtures that are available.  This can usually prevent people from fully learning and understanding the library before moving on to tooling etc.

It is a common misconceptiong that tooling is required to learn ReactJS.

Additional bleeding edge topics that often appear in React threads such as:
  • Inline styles

Tooling

ReactJS is platform agnostic which means I can use React with Rails, Node, Java, PHP or even in a plain HTML file.  Because React is so flexible a lot of tooling is often involved just to get started.  Today I will show you how to get started with ReactJS without any tooling at all.  We will build a production ready application using only ReactJS and node and nothing more.


ReactJS

First of all what is ReactJS?

ReactJS is a JavaScript library (not framework) that was created by Facebook for rendering data as HTML elements.

ReactJS is just the UI.  ReactJS is often used as the V in MVC as it makes no assumptions about the rest of your stack (although many experienced ReactJS devs will tell you it is also the C).

React does not make any assumptions about the rest of your stack.

Key Features

  • 1-way data flow
    • Properties are set as immutable data objects.  Data flows down and state keeps track of the current state of the component at any given time.
  • Virtual DOM
    • The DOM is preserved when writing ReactJS.  React keeps track of the DOM state in memory and diffs the DOM to determine which elements to update in the component.
  • JSX
    • A JavaScript extension that allows you to write an XML/HTML like syntax in JS

Why use React?

  • Speed!  
    • ReactJS is fast and delivers a better user experience
    • No detectable client side latency
  • Structured JS
    • No more spaghetti code in JS
  • Modern web experience
  • Happy developers
  • Server side rendering
  • Perfect match for ES2015

What About Other Solutions: Angular, Ember, etc...

Angular, Ember, and any other JavaScript frameworks are MVC frameworks.  ReactJS is just for the User Interface (Just the View layer).  ReactJS is a library not a framework.

React is a very small library.  

You can create React components in an existing app very quickly and easily without a full rewrite of the app.


Virtual DOM

One of the greatest advantages to using React is the Virtual DOM.  With the virtual DOM we let the data describe the UI instead of manually manipulating the DOM with libraries like jQuery.

The virtual DOM works by keeping the current state of the app at any given point in memory.  It then performs a DIFF agains the DOM.  If there are any differences they will be updated on the page to reflect the differences.  The virtual DOM only updates the changed part of the DOM not the entire DOM.  This makes it incredibly fast and incredibly easy to build a dynamic UI.

state and props


Data inside of React is managed by state and props.
State is the current component state for any given component.  When state changes the DOM will be diffed with the virtual DOM and any part of the component that has changed data will update in realtime without refreshing the browser.  In order to work with state we need a constructor and we need to initialize a state object.
import React from 'react';
import List from './List';
class TodoApp extends React.Component {
constructor(props) {
super(props);
this.state = { items: [] }
}
render() {
return (
<div>
<List items={this.state.items} />
</div>
)
}
}
export default TodoApp;
In the component above I am setting the initial state of items to and empty array and on line 13 I am passing the current state of items down to the child component List.
I am passing something called items down to List with a value of the current state of items.  Now inside of list I can call this.props.items to retrieve that array.  NOTE:  Props are immutable.  State should be treated as immutable unless setState is called (more on this later).

This is simply rendering a form but there are some interesting things going on here.  
First in the constructor we have to bind(this) to our addTodo function.  There are several other ways to handle binding but this one is the most straightforward.  Thea reason we need to bind is because we will want to use this.props inside of that function so we have to let the function know what `this` actually is.
Next we have a function which does nothing more than prevent the default action of submitting the form.

The form has a ref (we will get to that soon) and an onSubmit listener.  When the form is submitted the addTodo function will run.  The input also has a ref which we will get to shortly.
From AddTodo down to list we passed props so that our todo list items could be rendered.  We can also communicate back up the chain by sending a callback function.  Let’s assume that the work for addTodo will be handled in the parent compnent TodoApp so that TodoApp is the only component managing state.  This means we will want to pass a function down to the child so the child can call the function and communicate with the parent.
TodoForm.js
addTodo(e) {
e.preventDefault();
let item = this.refs.item.value;
this.props.addTodoItem(item);
this.refs.form.reset();
}
1.  Prevent form from being submitted
2.  Get the value out of the input this.refs.item returns the entire DOM node so .value needs to be called on the end.
3.  Call a callback function that will be passed down from the parent passing in the contents of the input field.
4.  Clear out the form.
Now for the callback function
src/component/TodoApp.js
import React from 'react';
import List from './List';
import TodoForm from './TodoForm';
class TodoApp extends React.Component {
constructor(props) {
super(props);
this.addTodoItem = this.addTodoItem.bind(this);
this.state = { items: [], id: 0 }
}
addTodoItem(name) {
let id = ++this.state.id;
this.setState({
items: [
{ name, id },
...this.state.items
],
id
})
}
render() {
return (
<div>
<TodoForm addTodoItem={this.addTodoItem} />
<List items={this.state.items} />
</div>
)
}
}
export default TodoApp;
Line 8 is binding this to the addTodoItem function because we will need access to state in that function.
state.items is now an empty array and state.id has been added since we don’t have a database we need to simulate unique id’s.
Line 21 has been updated to pass the callback function down to the child.

We give items a new value of the object that we just created and everything else that was already in the array.

Pass the onTodoClick callback down to the item which actually needs to come from TodoApp so that we can modify the data.

src/components/TodoApp.js
constructor(props) {
super(props);
this.addTodoItem = this.addTodoItem.bind(this);
this.onTodoClick = this.onTodoClick.bind(this);
this.state = { items: [], id: 0 }
}
onTodoClick(id) {
let items = this.state.items.map( item => {
if ( item.id === id ) {
return {
...item,
complete: !item.complete
}
}
return item;
});
this.setState({ items });
}
Bind onTodoClick in the consturctor.

The onTodoClick takes in an id of whichever items was clicked on.
It loops over all of the items in the state.  
If the item in the loop is not the item we are looking for then we just return the item.
If it is the one we are looking we return the item but override the complete property setting it to the opposite of whatever it was before we click.
Now when you run the app you should be able to click on items to mark them as completed or not completed.

case 'Active':

import React from 'react';
const filterLink = (text, current, cb) => {
if (text === current)
return (<span>{text}</span>);
return (<span onClick={() => cb(text)} style={{ textDecoration: 'underline', color: 'blue' }}>{text}</span>);
}
const Filter = ({ currentFilter, setFilter }) => (
<div>
{filterLink('All', currentFilter, setFilter)}
{' '}
{filterLink('Active', currentFilter, setFilter)}
{' '}
{filterLink('Complete', currentFilter, setFilter)}
{' '}
</div>
);
export default Filter;
This component has a helper function that returns a clickable “link” if it is not the current filter or a span if it is.
We simply call the helper function one time for each type of filter.

Lifecycle methods:

React has several lifecycle methods that are very useful when using a class:
componentWillMount() {
/*Invoked once, both on the client and server, immediately before the initial rendering occurs. If you call setState within this method, render() will see the updated state and will be executed only once despite the state change. */
}
componentDidMount() {
/* Invoked once, only on the client (not on the server), immediately after the initial rendering occurs. At this point in the lifecycle, you can access any refs to your children (e.g., to access the underlying DOM representation). The componentDidMount() method of child components is invoked before that of parent components.
If you want to integrate with other JavaScript frameworks, set timers using setTimeout or setInterval, or send AJAX requests, perform those operations in this method. */
}
componentDidUnMount() {
/* Invoked immediately before a component is unmounted from the DOM.
Perform any necessary cleanup in this method, such as invalidating timers or cleaning up any DOM elements that were created in componentDidMount.*/
}
componentWillUpdate(nextProps, nextState) {
/* Invoked immediately before rendering when new props or state are being received. This method is not called for the initial render.
Use this as an opportunity to perform preparation before an update occurs. */
}
componentDidUpdte(prevProps, prevState) {
/* Invoked immediately after the component's updates are flushed to the DOM. This method is not called for the initial render.
Use this as an opportunity to operate on the DOM when the component has been updated. */
}
componentWillReceiveProps(nextProps) {
/* Invoked when a component is receiving new props. This method is not called for the initial render.
Use this as an opportunity to react to a prop transition before render() is called by updating the state using this.setState(). The old props can be accessed via this.props. Calling this.setState() within this function will not trigger an additional render. */
}
shouldComponentUpdate(nextProps, nextState) {
/* Invoked before rendering when new props or state are being received. This method is not called for the initial render or when forceUpdate is used.
Use this as an opportunity to return false when you're certain that the transition to the new props and state will not require a component update.
*/
}
